# Magic Cube 4D

Interactive **3×3×3×3** (3⁴) hypercube in the browser, built with **Three.js**. The puzzle model and camera follow [Hyperspeedcube](https://github.com/HactarCE/Hyperspeedcube) conventions (facet-centered view, ±⅓ layer cuts, 4D→3D projection).

This repository also hosts the reproducible code for our IEEE Transactions on Games paper, *Structural State-Depth Calibration in the Four-Dimensional Rubik's Cube: Exact Distances, Scramble Bias, and Move Metrics*. The interactive app above is the game environment (Fig. 1 of the paper); the measurement-and-calibration pipeline lives in **[`paper/`](paper/)**.

## Paper / reproducibility

All experiments, tables, and figures in the paper are regenerated by the scripts in [`paper/`](paper/) with fixed seeds. See **[`paper/README.md`](paper/README.md)** for the full table/figure → script map. Quick start:

```bash
pip install sympy matplotlib numpy pillow
cd paper
python3 spheres.py            # sphere sizes, 3x3x3 validation (OEIS), diameter bounds  (Tables II,III,VII,VIII)
python3 certify.py --quick    # certified-ball exact depths + scramble bias            (Tables IV,VI)
python3 free_reduction.py     # exact free-reduction decomposition + cube corroboration (Table V)
python3 htm_scramble.py --quick  # half-turn-metric scramble bias                       (Table IX)
python3 generator.py --quick  # depth-targeted generator: accuracy, diversity, latency  (Tables X-XII)
python3 figures.py            # regenerate figures/
```

Drop `--quick` (and add `--deep` to `spheres.py`) for the paper's full sample sizes and the fingerprinted depth-5 / HTM depth-4 counts. The pipeline reproduces the ordinary 3×3×3 cube's published sphere sizes (OEIS A080602 / A080601) as a correctness check, and the certified-exact distance results depend only on the full-key radius-4 ball.

## Run locally

No build step — static HTML/JS modules.

```bash
python3 -m http.server 8765
```

Open [http://localhost:8765/index.html](http://localhost:8765/index.html).

## Features

- **3⁴ puzzle** — 81 cells, 8 face colors (+X/−X … +W/−W), animated quarter-turns
- **Scramble** — length 1–30; move history saved in `sessionStorage`
- **Solve** — reverses the recorded scramble (inverse of in-app twists); reliable when you scrambled with this app
- **Undo** — undo last twist
- **Solve log** — manual twists, solve playback, and errors with readable move labels
- **View controls** — drag to orbit; continuous spin on W axis, ZW plane, or 3D view; rotation speed slider
- **Sidebar** — puzzle controls on top; below that, solve log + guide (left) and face / twist tools (right)

## Controls

| Input | Action |
|--------|--------|
| **Drag** | Rotate 3D view (XZ / YZ) |
| **Shift + drag** | Rotate into W (XW / YW) |
| **Ctrl + drag** | ZW plane rotation |
| **Click cubie face** | Select twist face |
| **Select Face** buttons | Pick face axis (+X … −W) |
| **Layer** (outer / middle / inner) | Which slice turns along that face |
| **Plane A / B / C** | Which 2D rotation plane inside the facet (shallow / wide / deep) |
| **↺ CCW / ↻ CW** | Apply twist on selected face |
| **Scramble** | Random scramble (length from slider) |
| **Solve** | Undo scramble by replaying inverse moves |
| **Reset** | Solved state; clears history and log |
| **Undo** | Undo one recorded twist |
| **Recenter view** | Restore facet-centered camera |
| **Rotate W / ZW / continuously** | Auto-rotate view (camera only, not the puzzle) |
| **Stop rotation** | Stop auto-rotate |

View rotation never changes the puzzle — only the 4D projection matrix.

## Move notation (solve log)

Each twist is logged in plain language, for example:

`Back · outer · plane B (wide) · CCW`

That means: **Back** face, **outer** layer, rotation in **plane B (wide)**, **counter-clockwise** when looking at that face.

## Sidebar layout

| Area | Contents |
|------|----------|
| **Puzzle** | Scramble length, Scramble / Reset / Undo, view rotation, Solve, progress line |
| **Left column** | Solve log (top half), **Guide** (layer / plane / log help) |
| **Right column** | Select Face, **At a glance** (state, selection, move count), Twist Layer, keyboard **Controls** |

## Architecture

```
index.html          UI
css/style.css       Neumorphic layout
js/main.js          App wiring, history, solve log, selection summary
js/controls.js      Pointer drag → view rotation
js/engine/
  hyperpuzzle.js    81-slot state, twists, scramble
  renderer.js       Three.js cubies, edges, picking
  projection.js     HSC-style 4D→3D math, default view
  moves.js          Atomic moves, apply / invert (solver + verify)
  notation.js       MC4D macro parse, verbose formatMove
  puzzle-store.js   sessionStorage history / solutions
  solve-log.js      Step-by-step log panel
  solver.js         Inverse scramble, macro phases, search (tests / internal)
  macros.js         Beginner-method macro sequences
  puzzle-state.js   Solved checks, heuristics, state keys
```

Legacy files `js/hypercube.js`, `js/math4d.js`, `js/renderer.js` are superseded by `js/engine/*`.

Solver notes: [`js/engine/SOLVER.md`](js/engine/SOLVER.md).

## Limits

- **Solve** only guarantees restoration for positions reached via **Scramble** or twists recorded in `moveHistory`. Extra manual moves after a scramble require **Scramble** or **Reset** before Solve if history no longer matches the cube.
- The internal blind solver (macros + BFS / beam / IDA*) is heuristic and not exposed in the UI; it is not a general optimal 4D solver.
- Rendering uses shrunk cubies per visible facet rather than Hyperspeedcube’s exact CSG sticker meshes.

## Matching Hyperspeedcube exactly

A pixel-identical port would need Rust mesh export or a full CSG/WGSL pipeline. This repo implements the **same puzzle topology** (layers, three facet rotation planes, sticker tracking) and **same default camera** in JavaScript so you can iterate without a Rust build.
